# Refs 使用说明（给领域 Agent / 协作者）

本文定义本项目 refs（参考资料库）的使用方式，目标是让回答做到：

- **有依据**：结论/建议可追溯到明确来源
- **可控**：不把大量原文塞进上下文导致 token 膨胀
- **一致**：不同模型/不同人生成的输出仍保持相同“引用形状”

---

## 1. refs 与 skills 的分工与关联

- **skills**：负责“怎么做”（工作流、输出模板、检查清单）
- **refs**：负责“凭什么这么说”（证据、引用来源、定位信息）
- **关联方式（推荐）**：
  - 在 Skill 的流程中明确：回答前先查 `docs/refs/index.md`，挑选 1–3 条最相关 refs
  - 只将命中的**短摘录**（必要时）注入上下文
  - 输出末尾统一附 `## References`，按本说明的格式给出引用

> 经验法则：**同一请求尽量只启用 1 个主 skill**；refs 通过索引按需命中即可。

---

## 2. token 与“refs 越多越贵吗”

- refs 文件放在仓库里不会自动消耗 token。
- **只有被带进模型上下文的内容**（摘录/全文/工具检索结果注入）才消耗 token。

### 建议限额（可按团队实际调整）
- 每次回答：refs 命中 **≤ 3 条**
- 每条摘录：**≤ 8 行**（或 ≤ 600 字）
- 只在“需要佐证的结论点”处摘录，不要为每句话摘录

---

## 3. 如何新增一条 ref（写入 `docs/refs/index.md`）

### 3.1 必填字段（最小可用）
- `id`：唯一 ID（稳定不变，便于长期引用）
- `title`：标题
- `publisher`：作者/机构/期刊/组织
- `year`：年份
- `type`：类型（standard/whitepaper/paper/guideline/internal）
- `topic` + `keywords`：用于检索
- `source`：文件路径或链接（受控链接也可，但需注明访问约束）
- `locator`：章节/页码/段落号（越精确越好）
- `summary`：一句话摘要（用于筛选）

### 3.2 可选字段（推荐）
- `quote_hint`：建议摘录范围（例如“Clause 6.4 定义段”）
- `status`：`ready | placeholder | deprecated`

### 3.3 建议的 ref 文件存放
如果你希望把原文也纳入仓库（注意版权/合规）：
- 建议放到 `docs/refs/sources/`（可再按 `standards/`、`papers/` 分类）
- 大文件可只存“允许分享的节选/摘要”，原文用受控链接指向

---

## 4. 如何在回答中引用 refs（统一格式）

### 4.1 输出末尾统一 `## References`
每条引用建议包含：
- **REF id**
- **title / publisher / year**
- **locator**（章节/页码/段落）
- **与当前结论的关联一句话**

示例（模板）：

```text
- REF-XXXX：<title>（<publisher>, <year>），<locator>。用于支撑：<本回答中的结论点>。
```

### 4.2 什么时候需要“原文摘录”
仅在以下情况建议摘录 1 段短原文（并控制行数）：
- 争议点/边界点（例如 FuSa vs SOTIF 边界）
- 定量/定义性陈述（例如术语定义、公式、严格要求）
- 需要避免“凭感觉”误读的条款

---

## 5. 与 Rules 的配合（建议）

如果你希望团队输出更稳定，可以在 Rules 中加入“引用硬约束”，例如：
- 结论/建议必须附 refs（或明确标注“无可用引用”并降级为假设）
- 禁止在证据不足时下合规断言（与 FuSa 领域规则一致）

---

## 6. 常见坑与应对

- **坑：索引写得太长** → 只保留元数据与摘要，全文不要放进 `index.md`
- **坑：每次都注入大量 refs 原文** → 改为“索引检索 + 命中段落注入”
- **坑：引用不带 locator** → 要求至少到“章节/页码/段落”级别，否则难追溯
- **坑：占位条目长期不补齐** → 用 `status: placeholder` 标记，并在需要时优先补齐
