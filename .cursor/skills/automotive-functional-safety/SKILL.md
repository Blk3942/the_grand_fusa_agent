# 汽车功能安全（ISO 26262）工作流技能

## 使用方式

1. 先确认用户所处 **V 模型阶段**（概念、系统、硬件、软件、生产、运行等）及 **Item Definition（相关项定义）** 是否已提供。
2. 缺信息时**列出假设**与**需补充的最小信息集**，再给出建议，避免虚构客户项目细节。
3. 输出中显式区分：标准要求 / 常见行业实践 / 项目待定项。

## 核心工作流速查

### 1. Item Definition（相关项定义，ISO 26262-3）

- 边界、接口、运行环境、已知危害与约束；与外部要素的交互。
- 输出：清晰的范围与接口表，便于后续 HARA 不遗漏场景。

### 2. HARA（危害分析与风险评估）

- 危害识别 → 场景 → 危害事件 → **ASIL 等级**（严重度 S、暴露 E、可控性 C）。
- 给出 **安全目标（Safety Goal）** 及对应 ASIL；说明是否需 **safe state**。
- 提醒：ASIL 定级依赖场景假设，变更场景需重评。

### 3. FSC（功能安全概念）

- 从安全目标到 **功能安全要求（Functional Safety Requirements）**；容错、告警、降级策略。
- 与架构要素（系统/软件/硬件）初步映射。

### 4. TSC / 系统阶段（技术安全概念与系统设计）

- 技术安全要求、系统架构对安全机制的承载；冗余、监控、诊断覆盖的初步论证。

### 5. 软硬件阶段（概要）

- 软件：避免单点、内存保护、调度、编码规范、静态/动态测试与 ASIL 的匹配（参考 ISO 26262-6）。
- 硬件：随机硬件失效度量、诊断覆盖、FMEDA/FTA 等与 ASIL 的关系（参考 ISO 26262-5）。

### 6. 验证、确认与安全确认（validation / safety confirmation）

- 测试层级（单元、集成、系统、整车）、安全确认与 **确认评审（confirmation review）** 的差异。
- 强调证据：**测试规格、结果、可追溯矩阵**。
