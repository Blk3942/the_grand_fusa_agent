## ASIL：QM、A、B、C、D

- ASIL 由 HARA 的 **S/E/C** 组合导出；**QM** 表示质量管理即可、无需按该安全目标执行 ISO 26262 的全套要求（仍以组织流程为准）。
- **分解（decomposition）** 需满足标准前提（独立性、共存分析等），不可随意「拆 ASIL」。

## 与 SOTIF（ISO 21448）的边界（简述）

- **ISO 26262**：主要针对 **E/E 系统故障**导致的安全风险。
- **SOTIF**：主要针对 **无故障情况**下，由于场景不足、性能局限、误用等导致的危害。
- 实际项目常联合开展，但分析条目与证据分开管理更清晰。

## 常见误区

1. **把「功能多」等同于「ASIL 高」** — ASIL 来自危害事件与场景，不来自功能数量。
2. **忽略可控性 C 的场景定义** — 同一危害在不同交通场景下 C 可能不同。
3. **混淆 verification 与 safety confirmation** — 二者目的与责任人语境不同，勿混用术语。
4. **将 AI 生成内容直接作为认证证据** — 生成内容需经人工评审、与项目数据挂钩并归档。
